# PART 3 — KNOWN BUGS (fix before testing)

## BUG 1 — Feedback Analysis: JSON Parse Failure (Critical)

**Symptom** (seen in screenshot terminal):
```
→ Received 1372 chars from AI
→ Debug: raw 1372 chars. First 80: * Role: Analyst improving Flowr AI. * Task: Read conversation evidence a
! JSON parse failed: Unexpected token '*', "* Role: "... is not valid JSON
✓ Recovered JSON object: 0 plans
```
or:
```
X All parsing failed. Raw start: * Role: Analyst improving "Flowr AI". * Input: Evidence...
```

**Root cause:** The `analyze` route uses `gemini-1.5-flash` with a prompt that says "Respond ONLY with a valid JSON array" — but:
1. The current running config uses `gemma-4-31b-it` as backend model (seen in settings screenshot)
2. Gemma ignores the JSON-only instruction and outputs markdown prose starting with `* Role: Analyst...`

The code already has `raw.match(/\[[\s\S]*\]/)` extraction — but the model never emits `[...]` at all, just prose.

**Fix (in `/api/ai/brain/analyze/route.ts`):**
```ts
// Make the JSON instruction impossible to miss:
const analysisPrompt = `CRITICAL: Your response must be ONLY a valid JSON array starting with [ and ending with ].
No markdown. No explanation. No role description. Just JSON.

// ... rest of prompt ...

OUTPUT FORMAT — MANDATORY:
[{"topic":"...","title":"...","reasoning":"...","plan":"..."}]
START YOUR RESPONSE WITH [ AND END WITH ]`
```

Also: switch the analyze route to use `backend_model` from DB (like the manage route does) instead of hardcoding `gemini-1.5-flash`.

# PART 4 — SYSTEM DOCUMENTATION (for reference)

## Feedback Analysis Pipeline (Full Flow)

```
User selects messages in Feedback Logs
         ↓
"Send N to Analysis" button
         ↓
POST /api/ai/brain/analyze  { log_ids: [...] }
         ↓
SSE stream starts:
  → Creates bot_analysis_sessions row (status: running)
  → Loads targeted message_logs by log_ids
  → Loads all message_feedback (like/dislike)
  → Loads bot_brain_entries for context
  → Builds analysis prompt with disliked/liked samples
  → Calls Gemini (gemini-1.5-flash hardcoded, should use backend_model)
  → Parses JSON array from response
  → Inserts rows into bot_improvement_plans
  → Updates session (status: complete)
  → Sends { type: 'complete', plans: [...] }
         ↓
FeedbackClient receives log lines → shows terminal
         ↓
Plans appear on Routine page
```

**Current bug:** Step "Parses JSON array" fails when model returns prose instead of JSON.

## Routine → Brain Flow (Plan Accept)

```
Routine page shows pending plans
         ↓
User clicks Accept
         ↓
acceptPlan() in planActions.ts
  → topicToCategory(plan.topic) maps topic string → BrainCategory
  → addBrainEntry(category, title, plan)  ← CRASHES if category is old name
  → Updates bot_improvement_plans status = 'accepted'
  → Logs to activity log
         ↓
Entry appears in Bot Brain page
         ↓
User clicks "Sync Brain" in Bot Settings
         ↓
Compiled prompt rebuilt with new entry injected
```

## Web Search Pipeline

```
User message → classify as WEB_SEARCH
         ↓
chainRouter iterates WEB_SEARCH chain:
  [tavily-search (vault) → gemma models → duckduckgo (vault)]
         ↓
provider: 'vault', id: 'tavily-search'
  → runWebSearchChain(prompt)
     Stage 1: Tavily SDK search (advanced, 5 results)
     Stage 2: Gemini gemini-1.5-flash synthesizes answer from results
  → Returns synthesized string
         ↓
On Tavily failure → next model in chain (Google)
On all Google failures → duckduckgo-search fallback
```

## Image Generation Pipeline

```
User message → classify as IMAGE_GEN
         ↓
chainRouter iterates IMAGE_GEN chain:
  [pollinations → huggingface-stable-diffusion → cloudflare-workers-ai]
         ↓
Each provider returns Buffer | null
If string returned (refusal/description) → skip to next ← KEY LOGIC
         ↓
Confirmed working order (from screenshot):
  pollinations ✗ → HF stable diffusion ✗ → cloudflare ✓ → image returned
         ↓
Response type: 'photo', content: Buffer
```

## Brain Manager Chat Flow

```
User types in Brain Manager input (or clicks quick-action button)
         ↓
POST /api/ai/brain/manage  { message, history }
         ↓
SSE stream:
  → Loads all bot_brain_entries + bot_settings
  → Picks provider from backend_model setting
  → Streams response
  → Model wraps visible text in [ANSWER]...[/ANSWER]
  → Model puts actions in [ACTIONS_START]...[ACTIONS_END]
         ↓
Frontend receives:
  { type: 'text', content: chunk }   → displays in chat
  { type: 'actions', actions: [...] } → (action cards not confirmed built)
  { type: 'done' }
```

Quick-action buttons (seen in screenshot):
- **Analyze all** — sends "Analyze all brain entries and suggest optimizations — check for duplicates, overlaps, wrong categories, and possible merges."
- **Sort entries** — sends predefined sort prompt
- **Find duplicates** — sends predefined duplicates prompt
- **Suggest deletions** — sends predefined deletion prompt
